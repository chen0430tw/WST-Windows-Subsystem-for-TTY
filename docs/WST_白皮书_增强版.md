# WST 白皮书（增强版）

## 一、项目定义

**WST（Windows Subsystem for TTY）** 是一个运行在 Windows 宿主系统上的伪 TTY 子系统，目标是在不修改 Windows 内核的前提下，为 Windows 补上一层接近 Linux TTY 风格的可快速切换控制层。

它不是对 Windows 内核的替换，也不是对 Linux 虚拟控制台机制的直接复制，而是一种建立在 Windows 现有进程、控制台、终端、会话与桥接能力之上的次级系统层。  
WST 的核心目标，是让用户在 Windows 中获得一种更接近“系统级维护控制台”的体验：快速唤起、低干扰、可持续会话、可桥接 Unix 能力、可作为维护与开发入口。

从工程定位上看，WST 更接近：

- Windows 上的伪 TTY 子系统
- 本地维护控制层
- 面向开发者与系统玩家的次级操作界面
- 后续可扩展为 MicroVM / 独立桌面 / 宿主-客体混合控制平台的前置层

WST 第一阶段不追求成为“第二操作系统”，而是先成为一个**可用、可扩展、切换足够快的本地控制子系统**。

---

## 二、问题背景与设计动机

Windows 长期拥有成熟的图形桌面、Win32 API、控制台子系统、ConPTY、PowerShell、WSL、Cygwin 等多种组件，但在交互结构上，它并没有像 Linux 那样的原生虚拟控制台切换体验。  
Linux 用户可以通过 `Ctrl+Alt+F3` 一类机制快速切入纯文字控制台，获得脱离图形桌面的次级控制空间；而 Windows 虽然拥有终端、控制台、用户切换、恢复环境与多桌面对象机制，却缺少一种统一、轻量、常驻、面向开发与维护的“快速次级控制层”。

这就导致一个长期存在的空白：

- 普通终端太像应用，不像系统入口
- 图形桌面太重，不适合做极简维护切换
- WinRE 太偏恢复，不适合日常控制
- shell replacement 太重，侵入性太强
- 自定义壳层往往缺乏统一后端与会话持久层

WST 正是在这个空白中提出的。  
它试图回答的问题不是“如何把 Windows 变成 Linux”，而是：

**如何在 Windows 内部，补出一个具有 TTY 风格体验的可切换控制子系统。**

其设计动机主要来自四点：

第一，建立一个比普通 terminal 更像“系统入口”的控制层。  
第二，为 Windows 提供一个低干扰、可常驻、可快速唤起的维护和开发空间。  
第三，利用 `cygctl`、Cygwin、PowerShell、CMD、后续 WSL 等能力，形成统一后端接口。  
第四，为未来更深层的宿主-客体切换、独立桌面控制层、MicroVM 控制台预留架构空间。

## 三、术语表与边界定义

为了避免项目在后续讨论中发生概念漂移，本白皮书对以下术语作统一定义。

### 1. WST

WST 指 **Windows Subsystem for TTY**。  
它是一个运行在 Windows 宿主上的伪 TTY 子系统，而不是 Windows 内核替代物，也不是 Linux TTY 的直接复刻。

### 2. 伪 TTY 子系统

“伪 TTY 子系统”指的是：  
在宿主操作系统内部，构造出一个具有类 TTY 体验的次级控制层。  
它不要求宿主具备原生 VT/TTY 机制，但要求在用户体验与工程结构上，提供接近 TTY 的这些特性：

- 快速切换
- 会话可持续
- 输入输出纯文本导向
- 与图形桌面相对独立
- 适合作为维护与开发入口

### 3. 次级控制层

“次级控制层”是 WST 的核心概念。  
它位于普通 terminal app 与完整第二操作系统之间，既不只是一个黑窗口应用，也不以替换宿主系统为目标，而是作为一个长期常驻、可随时唤起的维护与控制空间存在。

### 4. Backend

Backend 指实际负责执行命令与提供环境能力的后端执行器。  
第一阶段优先支持：

- `cygctl`
- PowerShell
- CMD

后续可扩展：

- WSL
- daemon 型后端
- 远程控制后端
- MicroVM guest console

### 5. Native 层

Native 层指以 C 为主、直接对接 Windows 平台 API 的那一层。  
它负责热键、ConPTY、进程与句柄、窗口模式与平台脏活，不承担高层交互设计职责。

### 6. 不属于 WST 当前边界的内容

以下内容可以作为长期路线提及，但默认不属于第一阶段实现范围：

- MicroVM 正式接入
- 独立桌面对象接管
- shell replacement 全量替代
- 驱动级输入过滤
- 内核级切台
- 完整第二操作系统

## 四、系统目标与总体架构

WST 的总体目标可以压成一句话：

**在 Windows 宿主上构建一套具备快速切换、持续会话、统一后端和类 TTY 交互体验的伪子系统。**

为实现这个目标，WST 采用分层式结构。

### 1. 交互层

交互层由 Rust 实现，负责：

- 全屏或准全屏 TUI
- 输入框与输出区
- 状态栏
- 快捷命令面板
- 历史记录交互
- 会话可视化

这一层只负责“用户如何看到和操作 WST”，不直接承担底层 Windows 脏活。

### 2. 会话核心层

会话核心层负责：

- session 对象管理
- task 生命周期
- 命令分发
- 输出缓冲
- 重连与恢复
- scrollback 管理
- backend 状态抽象

这一层是 WST 的真正骨架。  
它决定 WST 是一个普通终端壳，还是一个具备“次级系统感”的控制层。

### 3. 原生平台层

原生平台层以 C 为主，负责对接 Windows 本体能力，例如：

- 全局热键
- ConPTY
- 进程创建与句柄继承
- 管道与 I/O 桥接
- 窗口置前 / 隐藏 / 全屏切换
- 未来的 desktop object / shell 接管实验接口

这一层的目标不是优雅，而是稳定、可控、易绑定。

### 4. 后端执行层

后端执行层是 WST 与各种命令执行环境之间的桥。

第一阶段优先支持：

- `cygctl`
- PowerShell
- CMD

后续可扩展：

- WSL
- daemon 型后端
- 远程控制后端
- MicroVM guest console

其中，`cygctl` 是第一阶段最重要的桥接组件。  
它不仅能为 WST 提供 Cygwin 命令执行能力，还能承担用户切换、提权桥接、stdio 接线和退出码传递等关键职责。  
因此，WST 与 `cygctl` 的关系不是简单的“调用外部工具”，而是“以 `cygctl` 为首个 Unix 能力后端”。

## 五、运行机制与用户体验设计

WST 的核心体验不是“启动一个程序”，而是“进入一个常驻的次级控制层”。

因此，它的运行机制采用如下思路：

### 1. 常驻与唤起

WST 推荐采用常驻守护进程 + 前端瞬时显示的模式。  
后台守护进程负责注册热键、管理 session、维护 backend 状态；前端界面在需要时被唤起到前台。

这意味着用户按下热键时，不是重新创建一个新环境，而是把已经存在的会话层瞬间切到前面。  
丝滑感来自“早已存在”，而不是“即时启动”。

### 2. 会话持续

WST 的会话不应随着前端关闭而销毁。  
用户可以：

- 唤起 WST
- 执行命令
- 暂时退出界面
- 再次唤起时看到原会话状态仍在

这类持续性是“像 TTY”的关键。  
如果每次唤起都从零开始，它就只是另一个 terminal app。

### 3. 命令模型

WST 采用双层命令模型：

一类是普通命令，直接交给后端执行。  
例如 `ls -la`、`uname -a`、`ps`。

另一类是 WST 内建命令，用于控制 subsystem 本身。  
例如 `:backend cygctl`、`:clear`、`:history`、`:status`。

这样可以在不强造完整 shell 语法的前提下，快速形成一个既有系统感又易扩展的交互模型。

### 4. 第一阶段体验目标

第一阶段的用户体验目标很明确：

- 一键唤起
- 黑底纯控制台风格
- 输入命令即执行
- 输出区稳定、低延迟
- 切出后会话不死
- 再切回时上下文仍在

只要做到这几点，WST 就已经具备“伪 TTY 子系统”的成立条件。

## 六、实现边界与阶段性范围

WST 必须严格按阶段推进，避免长期目标污染短期实现。

### 1. 第一阶段允许的最小目标

第一阶段默认只要求：

- `wst.exe` 可启动
- 进入全屏或准全屏 TUI
- 能输入命令
- 能调用 backend
- backend 至少支持 `cygctl`
- 输出可回显
- 错误输出可区分
- exit code 可见
- 有历史记录
- 有内建命令前缀，例如 `:status`

### 2. 第一阶段不要求的目标

以下内容默认不属于第一阶段验收范围：

- 完整 shell 语法
- 自己实现 Unix shell
- 真正复刻 Linux VT/TTY
- 真正操作系统级切台
- 图形桌面接管
- 自定义驱动
- 完整权限沙箱
- 完整多用户系统

### 3. 边界控制原则

任何新增设计都应先回答两个问题：

- 它是否直接服务于第一阶段伪 TTY 控制层成立？
- 它是否会显著增加当前实现复杂度而不提升当前可用性？

若答案分别为“否”和“是”，则该设计应后置。

## 七、开发协作规范

为防止项目在协作过程中失控，WST 必须采用明确的工作规范。

### 1. 不得擅自改项目定位

协作者不得把 WST 擅自改写成：

- 普通 terminal 模拟器
- GUI 终端工具
- 完整第二操作系统
- 纯 Linux 兼容层
- 与 `cygctl` 无关的独立项目

必须始终保持以下表述一致：

**WST 是 Windows 宿主上的伪 TTY 子系统 / 次级控制层。**

### 2. 必须坚持分层架构

任何设计、代码、目录结构都必须遵守下列分层原则：

- Rust UI 层：交互、面板、输入、状态展示
- Rust Core 层：session、task、buffer、history、命令调度
- Backend 层：`cygctl` / PowerShell / CMD / 后续 WSL 的统一抽象
- C Native 层：ConPTY、WinAPI、热键、句柄、窗口切换、平台脏活

不得把所有逻辑糊成一个 crate、一个文件或一个大函数。

### 3. 不得越阶段乱扩功能

第一阶段只允许围绕以下能力工作：

- WST 启动
- 全屏或准全屏 TUI
- 命令输入与执行
- `cygctl` backend 接入
- stdout/stderr/exit code 展示
- 历史记录
- 基础状态栏
- 热键唤起预留

以下内容不得擅自提前展开成重实现任务：

- MicroVM
- 独立桌面对象接管
- shell replacement
- 内核级接管
- 驱动级输入过滤
- 多 pane / 多 tab 完整系统
- 文件管理器
- 图形小部件系统

### 4. 代码输出必须文件化

凡是进入实现阶段，必须明确指出：

- 新增哪些文件
- 每个文件放什么
- 文件间依赖关系
- 哪个是入口
- 哪个需要用户补环境

不得只给“把这段塞进你的项目里”这种模糊答案。

### 5. 改动必须可回滚

每次代码输出必须尽量做到：

- 最小改动
- 明确增量
- 不破坏已有结构
- 能单独编译或至少静态可检查
- 容易 diff

不得为了“完整”而一次性推翻前面已经定下的目录与接口。

### 6. 不确定事项的处理规则

若无法确认某个实现细节，必须：

- 明确指出“不确定点”
- 给出保守默认方案
- 说明此方案为何适合作为第一版
- 不得伪装成已验证事实

## 八、对 `cygctl` 的集成规范

`cygctl` 是 WST 当前阶段最重要的后端桥接组件，因此必须单独立规范。

### 1. 默认原则

- 不绕过 `cygctl` 重新造一套同功能桥接
- 不擅自把 `cygctl` 降级为“仅可选工具”
- 第一阶段设计默认优先支持 `cygctl`

### 2. 接入原则

在涉及执行、桥接、命令转发时，应优先采用：

- 子进程调用 `cygctl`
- stdout/stderr 管道接回
- exit code 传回 session
- 统一封装成 backend trait 的实现

### 3. 不得做的事

- 不得擅自改写 `cygctl` 的定位
- 不得在没有明确要求的情况下重写 `cygctl`
- 不得把 WST 写死到只能跑 `cygctl`

最合理的关系应是：

**WST 提供 subsystem 外壳，`cygctl` 提供首个 Unix/Cygwin 后端。**

## 九、交付格式规范

WST 项目中的设计、代码与文档都应采用统一的交付格式。

### 1. 方案类交付

若任务属于规划、设计、蓝图、规范，交付内容应包含：

- 简短结论
- 3 到 5 个小节
- 每节只讲一个关键点
- 避免十几个小节的教材式目录

### 2. 代码类交付

若任务属于代码实现，交付内容应按以下顺序组织：

1. 本次新增/修改的文件列表  
2. 每个文件职责  
3. 完整代码  
4. 运行方式  
5. 验收方式

### 3. 修改类交付

若任务属于修、补、重构、接入，交付内容应说明：

- 是新增文件还是修改旧文件
- 保留了什么
- 替换了什么
- 为什么这样改
- 风险点在哪里

### 4. 文档类交付

若任务属于白皮书、规范、说明，交付内容应直接形成可保存为 `.md` 的成品，避免只讲写法而不给正文。

## 十、验收标准

WST 的交付结果必须满足结构、代码、工程、文档和交互五个维度的验收要求。

### 1. 结构验收

合格结果必须做到：

- 有明确目录结构
- 有明确模块边界
- 有入口文件
- 有依赖关系说明
- 有后续扩展空间

不合格表现：

- 全部逻辑混在一个 crate
- UI、backend、native 无分层
- 只给概念图，没有真实文件骨架

### 2. 代码验收

合格结果必须做到：

- 代码能看出职责划分
- 命名统一
- 不出现明显自相矛盾接口
- 至少具备静态可读性
- 运行流程清楚

不合格表现：

- 入口不清楚
- 函数名、结构体名随意漂移
- 前一段代码与后一段接口不一致
- 大量伪代码冒充成品代码

### 3. 工程验收

合格结果必须做到：

- 能说明如何启动
- 能说明如何测试
- 能说明哪些部分是 stub
- 能说明哪些部分依赖外部环境

不合格表现：

- 给完代码却没有运行方式
- 不说明 `cygctl` 如何接入
- 不说明哪些 API 只是占位

### 4. 文档验收

合格结果必须做到：

- 文档成品可直接保存
- 标题、层次、术语稳定
- 与 WST 定位一致
- 不把长期路线写成当前已经完成

不合格表现：

- 风格漂移
- 大量空泛口号
- 把“可能做”写成“已经实现”

### 5. 交互验收

合格结果必须做到：

- 说明用户如何进入 WST
- 说明用户如何执行命令
- 说明结果显示在哪里
- 说明退出与恢复方式

不合格表现：

- 只讲内部架构，不讲用户怎么用
- 不讲切换逻辑
- 不讲会话是否保留

## 十二、路线图与战略价值

WST 的开发不应一开始就冲向内核级侵入、桌面对象接管或完整虚拟控制台复制，而应分阶段推进。

### 第一阶段：伪 TTY 控制层

目标是做成一个可常驻、可热键唤起、可桥接 `cygctl` 的全屏控制子系统。  
这一阶段的成果是：

- WST 作为次级控制入口成立
- `cygctl` 成为 Unix 能力桥
- 会话、命令、历史、输出缓冲结构固定下来

### 第二阶段：宿主控制强化

在第一阶段稳定后，再加入：

- 更强的热键唤起
- backend 持久化
- 多会话
- PowerShell / WSL 并行后端
- 更细粒度的状态面板
- 桌面层切换实验接口

这一阶段，WST 会从“高级控制台”向“真实控制层”靠拢。

### 第三阶段：微系统化

当 WST 的会话与控制骨架足够稳定后，再考虑：

- 独立桌面对象控制层
- shell replacement 实验
- 本地 MicroVM 维护控制台
- 宿主-客体切换
- 快照 / 沙箱 / 回滚式维护模式

到这一阶段，WST 才真正可能成长为“Windows 上的次级微系统入口”。

---

## 第二阶段：宿主控制强化

### 一、阶段目标

第二阶段在第一阶段的基础上，将 WST 从”高级控制台”提升为”真实控制层”。

核心目标：
- 建立可靠的常驻机制
- 实现真正的会话持久化
- 支持多后端并行运行
- 提供更强的宿主集成能力

### 二、功能清单

#### 1. 强化的热键系统

**1.1 全局热键注册**
- 使用 Windows Hook 或 `RegisterHotKey` 实现全局热键
- 默认热键：`Ctrl+Alt+Space`（可在配置中修改）
- 热键冲突检测与自动规避

**1.2 热键行为**
- 首次按下：创建守护进程（如不存在）并显示前端
- 再次按下：切换显示/隐藏状态
- 前端关闭时：守护进程继续运行
- 守护进程退出：清理热键注册

**1.3 多显示支持**
- 记忆最后使用的显示器
- 支持跨显示器唤起
- 全屏状态在显示器间保持一致

#### 2. Backend 持久化

**2.1 持久会话**
- backend 进程在 WST 最小化时不退出
- 环境变量在会话间保持
- 工作目录记忆
- 后台任务继续运行

**2.2 会话恢复**
- WST 重启后恢复之前的 backend 状态
- 可选的会话快照与恢复
- 异常退出后的状态恢复机制

**2.3 Backend 生命周期管理**
```
┌─────────────┐
│  WST 启动    │
└──────┬──────┘
       │
       ▼
┌─────────────┐     ┌──────────────┐
│ 守护进程启动  │ ──▶│ Backend 启动  │
└──────┬──────┘     └──────────────┘
       │
       ▼
┌─────────────┐
│ 前端显示/隐藏 │ ◀─────────────────┐
└─────────────┘                   │
       │                          │
       ▼                          │
┌─────────────┐                   │
│ 用户交互      │                   │
└──────┬──────┘                   │
       │                          │
       ▼                          │
┌─────────────┐                   │

└──────┬──────┘                   │
       │                          │
       ▼                          │
┌─────────────┐                   │
│ Backend 继续 │                   │
│ (守护进程中)  │ ──────────────────┘
└─────────────┘
```

#### 3. 多会话支持

**3.1 会话模型**
- 每个会话对应一个独立的 backend 实例
- 会话 ID：64 位唯一标识
- 会话命名：用户可自定义会话名称
- 会话切换：通过快捷键或命令切换

**3.2 会话管理命令**
```
:session new <name>     # 创建新会话
:session list           # 列出所有会话
:session switch <id>    # 切换到指定会话
:session close <id>     # 关闭指定会话
:session rename <id>    # 重命名会话
```

**3.3 会话隔离**
- 每个会话独立的环境变量
- 每个会话独立的工作目录
- 每个会话独立的命令历史
- 会话间不共享进程空间

#### 4. 并行后端

**4.1 后端类型**

| 后端 | 用途 | 优先级 |
|------|------|--------|
| cygctl | Unix 命令执行 | P0 |
| PowerShell | Windows 系统管理 | P1 |
| CMD | 兼容性命令 | P1 |
| WSL | Linux 环境 | P2 |

**4.2 后端切换**
```
:backend cygctl      # 切换到 cygctl
:backend pwsh        # 切换到 PowerShell
:backend cmd         # 切换到 CMD
:backend wsl         # 切换到 WSL（未来）
```

**4.3 后端特性**
- 每个会话可独立选择 backend
- 支持跨后端命令传递
- 后端状态独立维护
- 后端异常隔离

#### 5. 增强状态面板

**5.1 状态栏扩展**
```
┌─ WST v0.2.0 │ cygctl │ Sess:3 │ Tasks:2 │ CPU:5% │ MEM:120MB │ [DEBUG] ─┐
└────────────────────────────────────────────────────────────────────────┘
```

**5.2 会话列表面板**
- 快捷键：`Ctrl+S` 或 `:sessions`
- 显示所有活动会话
- 支持快速切换
- 显示会话状态（运行中/空闲/异常）

**5.3 任务监控面板**
- 快捷键：`Ctrl+T` 或 `:tasks`
- 显示当前会话的所有任务
- 任务状态：运行/完成/失败
- 支持任务终止

#### 6. 桌面层切换实验

**6.1 实验目标**
探索在 Windows 上实现类似 Linux 虚拟控制台切换的可行性。

**6.2 技术路径**
- 使用 `SetThreadDesktop` 切换桌面对象
- 创建独立的安全桌面
- 探索全屏窗口在独立桌面上的行为

**6.3 实验范围**
- 仅作为可选功能
- 不影响主功能运行
- 需要用户明确启用

### 七、风险与限制

#### 7.1 已知风险
- 全局热键可能与其他软件冲突
- 守护进程可能增加系统资源占用
- 多会话可能增加内存消耗

#### 7.2 当前限制
- 桌面层切换仅在实验阶段
- 会话快照功能有限
- 跨后端命令传递需手动处理

## 结语

WST 的意义，不在于复制 Linux 的 TTY 机制本身，而在于为 Windows 建立一个长期缺位的控制层中间态：
它介于普通 terminal 和完整第二操作系统之间，既不等于简单应用，也不强行入侵内核，而是以 subsystem 的方式补足 Windows 在”快速维护控制体验”上的结构空白。

从产品角度看，WST 可以成为开发者、系统玩家和维护型用户的本地控制入口。
从技术角度看，WST 则是一个可逐步演化的敲门砖：先做伪 TTY，再做宿主控制层，最后再决定是否走向真正的微系统化。

而从工程协作角度看，本白皮书同时承担了**定位文档、实现边界文档、协作规范文档与验收基线文档**的作用。
后续任何设计、代码与文档输出，都不应偏离本白皮书所规定的边界、节奏与验收标准。
